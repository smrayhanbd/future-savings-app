### Phase 1 â€” Schema: double-entry backbone

**File:** `prisma/schema.prisma`

- Add **`JournalEntry`** (voucher header): `id`, `voucherNo` (unique, e.g. `JV-0001`), `entryDate`, `voucherType` enum (`JOURNAL`/`RECEIPT`/`PAYMENT`/`CONTRA`), `narration`, optional `referenceNo`, `memberId?`, `status` enum (`DRAFT`/`POSTED`), `totalDebit`/`totalCredit` `Decimal`, `createdAt`, `updatedAt`, relation to `JournalLine[]`.
- Add **`JournalLine`** (posting line): `id`, `journalEntryId`, `accountId` (FK â†’ `Account`), `debit`, `credit` `Decimal(12,2)`, `memo?`. Composite ensures each line posts to one `Account`.
- Extend **`Account`**: add `taxDeductible Boolean @default(false)`, `reconciledBalance Decimal?`, and a counter field is unnecessary (computed at read time).
- New enums: `VoucherType`, `JournalStatus`.
- **No breaking changes** to existing `Account` fields â€” purely additive. Existing `currentBalance` will be recomputed/maintained by postings.

Since this project uses `prisma db push` (no `migrations/` dir), the user will run `npm run build` (which runs `prisma generate`) or `npx prisma db push` to sync.
