## Overview

Two deliverables: (1) a new **Organization Info** settings page backed by an extended `Organization` model, and (2) a dedicated **printable Money Receipt** route for deposit & withdrawal transactions that renders a SaaS-grade voucher with org branding and active bank accounts. Both reuse existing patterns (singleton settings, Cloudinary upload, native `window.print()`, print-CSS conventions).

---

## Phase 1 — Extend the `Organization` model (Prisma)

The `Organization` model exists (`schema.prisma:13`) but only has `name` + `logo` and is unused. I'll extend it with all fields you approved, then convert it to a singleton pattern (matching `SiteContent`'s `id: "singleton"` convention).

**Edit `prisma/schema.prisma`** — extend `Organization`:
```prisma
model Organization {
  id            String   @id @default("singleton")  // singleton — one row
  // Identity
  name          String
  logo          String?                  // Cloudinary URL (uploaded via settings page)
  tagline       String?
  description   String? @db.Text
  // Contact + Address
  email         String?
  phone         String?
  website       String?
  addressLine   String?
  city          String?
  district      String?
  postalCode    String?
  // Legal / Registration
  regNo         String?                  // society/cooperative registration no.
  licenseNo     String?
  tradeLicenseNo String?
  establishedYear String?
  // Social (rendered as small icons on the receipt footer)
  facebook      String?
  whatsapp      String?
  youtube       String?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  members   Member[]
}
```

Then **`npx prisma db push --skip-generate`** (Supabase shadow-DB workaround from before) + **`npx prisma generate`** (after stopping the dev server).

## Phase 2 — Organization Info settings page

**Mirror the `SiteContent` pattern** (`app/dashboard/settings/site-content/` + `app/actions/site.ts`).

**New file `app/actions/organization.ts`** — server action `updateOrganization(formData)`:
- Reads all org fields from FormData
- Uploads the logo file via the existing `uploadImage(file)` helper from `lib/cloudinary.ts` (same pattern as `site.ts:39`)
- `prisma.organization.upsert({ where: { id: "singleton" }, update: {...}, create: { id: "singleton", ...} })`
- `revalidatePath` on every page that renders org info (receipts, landing, ledgers) + `redirect`

**New route `app/dashboard/settings/organization/`:**
- `page.tsx` (server) — `prisma.organization.findUnique({ where: { id: "singleton" } })`, fall back to empty values. Auth via `getCurrentUser()` + `isSuperAdmin` (mirroring `approval-limits/page.tsx`).
- `OrganizationForm.tsx` (client) — grouped form sections:
  - **Identity**: Name, Logo upload (preview + change), Tagline, Description (textarea)
  - **Contact & Address**: Email, Phone, Website, Address Line, City, District, Postal Code
  - **Legal / Registration**: Reg No, License No, Trade License No, Established Year
  - **Social Links**: Facebook, WhatsApp, YouTube
- Submit button → builds FormData → calls `updateOrganization`.

**Helper `lib/organization.ts`** — `getOrganization()` async helper that reads the singleton row and returns a plain serializable object (or sensible defaults: `{ name: "Future Savings Foundation", ... }`). Used by the receipt page, member ledger, account ledger, and any future consumer. Centralizing avoids each consumer re-querying and duplicating fallback logic.

## Phase 3 — Money Receipt route (the core deliverable)

**New dedicated route `app/dashboard/receipts/[transactionId]/`:**

This is a clean, print-optimized page. The URL is shareable and avoids print-CSS conflicts with the admin chrome.

### `page.tsx` (server component)
Loads everything the voucher needs in parallel:
- The `Transaction` with `member` (fullName, memberNo, phone, photoUrl) + `cashAccount` + `journalEntry.lines.account`
- `getOrganization()` (the singleton org info)
- Active `BankAccount` rows (with their COA + payment method) — for the "deposit in future" reference block
- Auth via `getCurrentUser()` (any authenticated user can print a receipt)

Only renders for `status === "APPROVED"` transactions (DRAFT/PENDING have no posted voucher). Returns 404 otherwise.

Rejects types other than `DEPOSIT` and `WITHDRAWAL` (the two you specified). CHARGE/INCOME_DISTRIBUTION route to the same template later if you want — for now, 404 with a clear message.

### `MoneyReceipt.tsx` (client component)
A SaaS-grade voucher layout. Print-optimized via `@media print` (Phase 4). Sections:

**Header band** (org branding):
- Organization **logo** (left) + **name** + tagline
- Right side: org address line, phone, email, reg/license no.
- Document title: **"MONEY RECEIPT VOUCHER"** for deposits, **"WITHDRAWAL VOUCHER"** for withdrawals (distinct title + accent color — green for deposit, rose for withdrawal)

**Transaction meta row** (two columns):
- Left: Voucher No (e.g. `TR-DEP-000001`), Voucher Type (RECEIPT/PAYMENT), Transaction Date, Approved Date
- Right: Payment Method (Cash/Bank/Mobile with label), Reference No (cheque/slip no)

**Payee/Payer block**:
- Member photo + name + member no + phone
- "Received with thanks from" (deposit) / "Paid to" (withdrawal) framing

**Amount block** (the focal point):
- Large amount in figures: **৳ X,XXX.00**
- Amount in words: "Taka One Thousand Five Hundred Only" (English number-to-words — small inline helper, no dependency)
- Purpose line (the breakdown.collectionTypeName / subType label)

**Accounting voucher table** (Dr/Cr):
- Reuses the journal lines data already on the transaction
- Columns: Account Code, Account Name, Debit, Credit, with Total row

**Active bank accounts block** (the "deposit in future" reference you asked for):
- A compact table listing each active `BankAccount`: Account Name, Bank/Provider, Method badge, Account/Wallet Number, Branch
- Heading: "For Future Deposits — Use Any of the Following Accounts"
- Only shown on the **deposit** receipt (not withdrawal)

**Signature block** (three columns):
- Received By / Authorized By / Account Member signature
- Each with a line + label + "Date: ___"

**Footer**:
- Org website, social icons (Facebook/WhatsApp/YouTube as text links)
- A generated system line: "This is a computer-generated voucher. Voucher No {voucherNo} · Generated on {now}"

### Print behavior
- Page has an on-screen toolbar (Back + Print buttons) marked `.no-print`
- The voucher itself is marked `.receipt-print-area`
- On `window.print()`, the `@media print` rules (Phase 4) hide the app shell and toolbar, show only the voucher, force white background, collapse borders for paper.

## Phase 4 — Print CSS (append to `app/globals.css`)

Extend the existing print block (lines 146-182) with receipt-specific rules, mirroring the `.portal-print-area` convention:

```css
@media print {
  /* Hide everything by default when printing a receipt */
  .receipt-no-print { display: none !important; }
  .receipt-print-area { display: block !important; }
  body { background: #fff !important; }
  .money-receipt { color: #111 !important; padding: 0 !important; box-shadow: none !important; }
  .money-receipt table { width: 100% !important; border-collapse: collapse !important; }
  .money-receipt th, .money-receipt td { border: 1px solid #ddd !important; background: #fff !important; }
  /* Avoid breaking the voucher across pages */
  .money-receipt, .money-receipt .signature-row { break-inside: avoid; }
  @page { margin: 12mm; }
}
/* On screen the print area renders normally (receipt is the main content of its route). */
```

## What I will NOT touch
- The approval/posting engine — receipts are read-only views over already-posted transactions.
- The member-facing portal — this is admin-side only (the portal has its own `.portal-print-area` statement).
- PDF generation libraries — sticking with native `window.print()` (consistent with the rest of the codebase; users can "Save as PDF" from the print dialog).

## Notes
- The `Organization` model will be converted to a singleton (`id: "singleton"`). Existing `Member.organizationId` relation still works if anyone links a member to an org; the singleton just becomes the default "our org" record.
- The amount-in-words helper will be a small English BDT converter (no external dependency). "Taka X Only" format.
- Receipt is **print-only optimized** but still looks good on-screen (white card on the page) so admins can preview before printing.
