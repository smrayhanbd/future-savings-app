# Plan: Mail Server & SMS Service API Settings

## Goal
Build the two missing settings pages the sidebar already links to — `/dashboard/settings/mail` and `/dashboard/settings/sms` (currently 404) — backed by DB-stored, encrypted provider configuration, with a templates editor, Test/Send-Test actions, and a settings audit log. The existing `sendEmail` / `sendSMS` functions are refactored to read the active provider from the DB instead of hardcoded env values, **without breaking any of the 13 call sites**.

## Scope (per user answers)
- **In scope:** All provider configs (SMS: BulkSMSBD, SendMySMS, SSLWireless, Twilio, Custom HTTP; Email: Gmail, Custom SMTP, SES, Resend, Mailgun, Brevo, Microsoft 365), AES-256-GCM secret encryption, templates editor, test connection / send test, settings audit log.
- **Out of scope (deferred):** Queue tables, logs tables, balance-fetch card, error-code mapping UI, retry/queue workers. (The `sendEmail`/`sendSMS` return contract stays unchanged so a queue can be layered on later.)

Templates: stored as a JSON column on each settings row? **No** — cleaner as a small dedicated model so the editor can CRUD them independently:

```prisma
model MessageTemplate {
  id        String   @id @default(uuid())
  channel   String   // "EMAIL" | "SMS"
  key       String   @unique // e.g. "MEMBER_WELCOME", "DEPOSIT_RECEIVED"
  name      String
  subject   String?  // email only
  body      String   @db.Text // SMS plain text or email HTML; supports {{vars}}
  variables String?  // comma list of supported vars, for reference
  updatedAt DateTime @updatedAt
}
```

**Migration:** I edit `schema.prisma` only. User runs `npx prisma migrate dev --name mail_sms_settings`.

## 2. New library files

### `lib/crypto.ts` (new)
Small AES-256-GCM helper using `node:crypto`. Key derived from a new env var `ENCRYPTION_KEY` via `scryptSync` → 32 bytes. Exports `encrypt(plain): string` (returns `iv:ciphertext:tag` base64-joined) and `decrypt(payload): string`. Throws clearly if `ENCRYPTION_KEY` is missing. Falls back to `NEXTAUTH_SECRET` only if `ENCRYPTION_KEY` is unset (with a logged warning), so the feature works even before the user adds the new var.

### `lib/mail/providers.ts` (new) + rewrite `lib/email.ts`
- `lib/mail/providers.ts`: pure functions that build a transport per provider and a `sendMail` call. One exported `getMailTransport()` that reads the active `MailSettings` row, decrypts secrets, and returns a configured nodemailer transporter (SMTP providers) or a Resend-style sender object.
- **Rewrite `lib/email.ts`** to keep the **exact same signature** `sendEmail(to, subject, html): Promise<void>` so all 5 email call sites (`approval.ts`, `auth.ts`, `finance.ts`, `meeting.ts`, `member.ts`, `transactions.ts`, `specialWishes.ts`) keep working unchanged. Internally it now: reads the singleton → builds transport → sends → on no-config/missing-secret falls back to the **legacy env-based Gmail transporter** so nothing breaks before the admin configures a provider. Adds `testMailConnection()` and `sendTestEmail(to, subject, body)` used by the settings page.

### `lib/sms/providers.ts` (new) + rewrite `lib/sms.ts`
- `lib/sms.ts` keeps the **exact signature and return shape** `sendSMS(to, msg): Promise<{ status: "OK" | "ERROR"; response?: string }>` — this is load-bearing for 5 callers that branch on `smsRes.status` and use `smsRes.response` (`approval.ts:76`, `member.ts:256`, `finance.ts:321`, `transactions.ts:638,707`, `specialWishes.ts:141`). Internally dispatches by active provider; falls back to the **legacy SendMySMS env path** if no DB config exists.
- `lib/sms/providers.ts`: one `send` function per provider (bulksmsbd, sendmysms, sslwireless, twilio, custom), each normalizing the response to `{status, response}`. Twilio uses its REST API via `fetch` (no SDK needed — keeps deps unchanged).
- Adds `testSmsConnection()` and `sendTestSms(to, msg)`.

### `lib/templates.ts` (new)
`renderTemplate(key, vars)` — load a `MessageTemplate` row by key, `{{var}}`-replace, return `{subject, body}` or `null` if not found (callers that don't use templates yet are untouched — this is opt-in for new code and the editor). Seed defaults on first run.

## 3. Server actions — `app/actions/messaging.ts` (new)

All `"use server"`, all guarded with `requireSuperAdmin(await getCurrentUser())` (the existing settings actions skip this — but these store secrets and write audit rows, so they must self-guard, per the server-actions doc's security section).

Actions:
- `saveMailSettings(formData)` — upsert singleton, encrypt secrets only when changed (blank field = preserve existing, mirroring `updateOrganization`'s logo-preservation trick), write `SettingsAuditLog`.
- `saveSmsSettings(formData)` — same pattern.
- `testMailConnection()` / `sendTestEmail(formData)` — call `lib/email`, return `{ok, message, ms}` for the UI.
- `testSmsConnection()` / `sendTestSms(formData)` — same.
- `saveTemplate(formData)` / `deleteTemplate(id)` — CRUD for `MessageTemplate`, audit-logged.
- All redirect/revalidate the relevant settings path.

## 4. Settings pages (the actual deliverable the sidebar points at)

### `app/dashboard/settings/mail/page.tsx` (new)
Server component. `getCurrentUser()` + `isSuperAdmin` guard (redirect pattern, matching `bank/page.tsx`). Loads `MailSettings` singleton + templates where `channel="EMAIL"`. Passes serializable props to the client form.

### `app/dashboard/settings/mail/MailSettingsClient.tsx` (new)
Client component. Sections (matching the spec layout):
1. **Provider picker** — card grid (Gmail / Custom SMTP / Amazon SES / Resend / Mailgun / Brevo / Microsoft 365). No radio component exists in `components/ui/`, so I build a styled button-card row (consistent with the bank page's method groups).
2. **Provider-specific fields** — only the selected provider's fields render. Secrets rendered masked (`••••••`) with a Show toggle; blank-on-save = keep existing.
3. **Common settings** card (Display Name, Reply-To, Retry, Timeout, Daily/Per-minute limits, Status switch).
4. **Test Email** card — recipient + subject + message + Send button, shows ✓/❌ + time-taken.
5. **Email Templates** editor — table of templates (Welcome, Password Reset, OTP, Deposit Received, …) with edit dialog using the existing `RichTextEditor` component for HTML body + a variables hint.
6. **Audit log** card — last ~10 `SettingsAuditLog` rows for `section="MAIL"`.

### `app/dashboard/settings/sms/page.tsx` + `SmsSettingsClient.tsx` (new)
Same structure, SMS providers (BulkSMSBD / SendMySMS / SSLWireless / Twilio / Custom HTTP). Custom HTTP provider has the full field set from the spec (auth type, body type, phone/msg/sender/apikey params, success field/value). Includes **Common Settings** (country code, phone format, URL-encode switch, retry, status) and the **Send Test SMS** card and **SMS Templates** editor and **audit log** card.

Both pages reuse existing UI primitives only: `Card`, `Input`, `Label`, `Select`, `Switch`, `Checkbox`, `Button`, `Table`, `Dialog`, `Tabs`, `Badge`, and `RichTextEditor` / `Textarea`. No new shadcn installs.
