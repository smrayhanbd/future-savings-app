## Fix: Registration duplicate email/phone popup + red-focused fields

### Root cause
1. **`<Toaster />` is never mounted** anywhere in the app (`app/layout.tsx`, `Providers.tsx`, and all nested layouts contain no `<Toaster`). The form's `toast.error(...)` logic is correct, but Sonner has no renderer, so the popup is silently dropped. This affects **all 9 files** that call `toast.*()`.
2. **Phone is not unique in the DB** (`prisma/schema.prisma`: `email @unique`, but `phone` only `@@index([phone])`). So duplicate phones are created silently today — the DB can't catch them. Needs an explicit pre-insert check.
3. **`RegisterForm.tsx` has no per-field error state / refs**, so it can't highlight or focus email/phone.

### Changes

**1. `components/Providers.tsx`** — Mount `<Toaster />` once at the app root (fixes toasts app-wide).
- Import `{ Toaster }` from `@/components/ui/sonner` and `toast` from `sonner`.
- Render `<Toaster position="top-center" richColors closeButton />` as a sibling of `{children}` inside `SessionProvider`. (`top-center` matches the "popup" feel the user described; `richColors` makes error toasts red.)

**2. `app/actions/member.ts`** — Make `registerMember` (and the `addMember` public path) detect both email **and** phone duplicates and return a structured error.
- In `addMember`, **before the upload/transaction**, add a `prisma.member.findFirst({ where: { OR: [{ email }, { phone }] } })` check. If a match exists, determine which (email, phone, or both) and `throw new Error(...)` with a typed message. (Done before uploads so no files are sent for a rejected request.)
- Wrap the `P2002` catch so its message is consistent with the new one.
- In `registerMember`, return `{ error: <message> }` for: the new pre-check, `P2002`, or any "already exists" message. Return `{ success: true }` otherwise (unchanged).
- *(Optional, since `phone` is intentionally non-unique per the schema — `emergencyPhone` etc. reuse phone-like values — I will **not** add `@unique` to `phone`. The explicit pre-check is the right call and avoids a risky migration.)*

**3. `app/register/RegisterForm.tsx`** — Surface the error as a red, focused email/phone field (in addition to the now-visible toast).
- Add `errors` state (`{ phoneNumber?: string; emailAddress?: string }`) and two `useRef`s for the inputs.
- When `registerMember` returns `{ error }`: parse the message to decide which field(s) to flag (email / phone / both), set `errors`, call `.focus()`/`.scrollIntoView()` on the first offending input, then show the `toast.error(...)`.
- Clear the relevant error on input change (`handleInputChange`).
- Apply conditional classes to the two inputs: when `errors.X` is set → `border-red-500 focus:ring-red-500/20 focus:border-red-500`, else the existing `inputClass`. Add a small red `<p className="text-red-500 text-xs mt-1">` message under each.

### Not changing (out of scope / intentional)
- The Add Member and Edit Member forms — they have their own (broken) error handling; this task is the public registration page only. (Mounting `<Toaster />` will quietly improve them too.)
- The `phone @unique` DB constraint — risky migration for a non-unique-by-design field. Pre-check covers it.
- No new dependencies (sonner already installed).

### Verification
- Submit with an existing **email** → red toast + email field red & focused.
- Submit with an existing **phone** → red toast + phone field red & focused (previously: silent success — now caught).
- Submit with **both** existing → message names both, both fields red, focus on email.
- Submit with a **new** email+phone → still routes to `/register/success` (no regression).
- Lint/typecheck via `npm run lint`.
